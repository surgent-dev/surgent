## Current flow

### 1. Sandbox bootstrap

When Surgent provisions a project sandbox:

1. It creates the sandbox with:
   - app client env vars
   - `SURGENT_API_KEY`
   - `SURGENT_BASE_URL`
   - `OPENCODE_API_KEY`
   - `OPENCODE_BASE_URL`
   - `OPENCODE_CONFIG_DIR`
2. It reads `surgent.json` from the generated app repo.
3. If present, it runs `scripts.init`.
4. If present, it starts `scripts.dev` under PM2.
5. It clones or updates the opencode config repo into `OPENCODE_CONFIG_DIR`.
6. It starts `opencode serve --hostname 0.0.0.0 --port 4096`.

### 2. Opencode harness shape

The opencode config is external to the main Surgent repo.

Important characteristics:

- Global instructions are very small: `STYLE_GUIDE.md` only.
- Convex is exposed through a remote Surgent MCP endpoint, not directly through Convex CLI instructions.
- Payments are exposed the same way.
- The orchestrator is UI-first and explicitly backend-later.
- The orchestrator tells agents not to send users to convex.dev or external dashboards.

### 3. Current Convex MCP surface

Current Surgent Convex MCP tools are:

- `create_project`
- `delete_project`
- `set_env_vars`
- `list_env_vars`
- `call_query`
- `call_mutation`
- `read_logs`

Notably absent:

- no `insights`
- no deployment creation/selection tools
- no env cloning/promotion tools
- no explicit local/anonymous init tool
- no function-spec or schema introspection tool

### 4. Current Convex mental model inside the harness

The harness currently assumes:

- create a cloud Convex project via `convex_create_project`
- sync Convex after backend changes via `dev-run { syncConvex: true }`
- frontend imports from `@convex/api` only after sync
- auth keys are auto-generated by Surgent during project creation
- users should use Surgent's embedded database tab when a UI is needed

This is a coherent model, but it predates the important 1.32-1.34 additions.

## What Convex 1.32-1.34 changes for this system

## 1.32

- `npx convex insights`
- insights MCP tool
- project-local `.convex` local backend state
- `maximumRowsRead` / `maximumBytesRead`
- deployment region choice for new deployments

## 1.33

- `npx convex init`
- `CONVEX_AGENT_MODE=anonymous`
- much better `env set` / `env list` workflows
- `run` and `env set` auto-start local backend

## 1.34

- `convex ai-files`
- `deployment create`
- `deployment select`
- `--deployment` targeting across commands

## Most important interpretation for Surgent

These releases do **not** mean the opencode harness should suddenly become CLI-first.

They mean Surgent can expose a better Convex operating model **through its own MCP/tools/UI**, while preserving:

- embedded dashboard UX
- agent-first workflows
- no external account/dashboard dependency
- current UI-first product philosophy

## Current gaps

### Gap 1: No health/diagnostics layer above logs

Current state:

- Agents can read raw Convex logs.
- Agents cannot ask Convex for high-level health signals like OCC conflicts or resource limit issues.

Impact:

- Agents over-index on code bugs when the problem may be contention, hot documents, missing indexes, or resource pressure.

### Gap 2: No deployment-aware Convex workflow inside MCP

Current state:

- Deployments are created/provisioned by Surgent internals.
- Agents do not get first-class tools for branch/staging/preview deployment flows.

Impact:

- Deployment targeting remains partly implicit.
- Harder to productize branch-bound Convex environments.

### Gap 3: Harness docs are still on the old sync/init model

Current state:

- Skills mention `convex_create_project` and `dev-run syncConvex: true`.
- They do not mention anonymous agent mode, init, region selection, env promotion, or new deployment commands.

Impact:

- Agents are not taught the best current Convex operating patterns.

### Gap 4: No deliberate position on local Convex mode

Current state:

- Surgent is effectively cloud-Convex-first.
- Local `.convex` behavior from 1.32 is not exploited.

Impact:

- Missed opportunity for ephemeral, no-login, branch-local Convex development.
- But this is optional, not mandatory.

### Gap 5: No policy for Convex AI files

Current state:

- Opencode already has its own style guide, orchestrator, coder, and skills.
- Convex AI files are not integrated.

Impact:

- Good: no instruction collisions today.
- Bad: no deliberate use of Convex's newer agent-specific context.

### Gap 6: Region support is missing from the actual platform provisioning wrapper

Current state:

- Surgent's worker-side deployment create wrapper sends `{ type }`.
- It does not expose region choice.

Impact:

- EU deployment support is not modeled in platform UX or provisioning policy.

## Proposed future flow

## Guiding rule

Do not expose raw new Convex CLI concepts directly to end users.

Expose them through:

- new MCP tools
- new opencode skill guidance
- selective worker automation
- embedded Surgent UI affordances

## Proposed flow A: Current default cloud flow, upgraded

This should be the default flow for most Surgent projects.

1. User asks for an app.
2. Orchestrator delegates UI-first work.
3. `dev-run` restarts preview so the user sees progress.
4. When backend is needed, `convex_create_project` provisions the cloud project as today.
5. Surgent persists:
   - project ID
   - deployment names
   - deploy keys
   - env vars
6. Agents use new MCP tools to:
   - inspect deployment health
   - clone/promote env vars
   - create/select branch deployments
   - inspect available deployments
7. User continues working entirely in Surgent UI and embedded tabs.

### New MCP tools I suggest for this flow

1. `insights`
   - input: `env`
   - output: summarized OCC/resource-limit findings

2. `list_deployments`
   - input: project/context
   - output: development, production, previews, refs, URLs, default flags

3. `create_deployment`
   - input: `type`, `ref`, optional `region`, optional `select`
   - output: deployment name, URL, selected status

4. `select_deployment`
   - input: deployment/ref
   - output: active sandbox target updated

5. `clone_env_vars`
   - input: source env/deployment, target env/deployment, optional overwrite mode
   - output: copied keys summary

6. `promote_env_vars`
   - input: dev -> prod or deployment A -> deployment B
   - output: changed keys summary

7. `function_spec`
   - input: env
   - output: function list/shape

8. `call_action`
   - current MCP only supports query and mutation
   - adding action support would reduce a real gap

### Why this is the best default

- Fits current Surgent architecture
- Fits current opencode orchestrator philosophy
- No user-visible CLI complexity
- Gives agents the missing operational tools

## Proposed flow B: Optional local anonymous Convex mode

This should be optional, for special templates or advanced agent sessions.

1. Sandbox starts.
2. Agent chooses local Convex mode.
3. Surgent runs an internal tool or script equivalent to:
   - `CONVEX_AGENT_MODE=anonymous`
   - `convex init`
   - local env seed
   - local sync/dev
4. Local backend state lives in repo-local `.convex`.
5. Each worktree/sandbox stays isolated.
6. If the project needs to graduate to cloud-hosted dev/prod, Surgent promotes it into managed cloud deployments.

### Why I would not make this the default yet

- Current harness is already built around cloud project provisioning and embedded cloud dashboard access.
- Local mode is more useful for ephemeral prototyping, branch experiments, or cost-sensitive early scaffolding.
- It introduces another mode that the orchestrator and skills would need to model carefully.

## Suggested harness changes

### Orchestrator

Add explicit guidance:

- after Convex deployment issues, use `insights` before blaming app code
- for staging/branch work, create/select a deployment rather than overloading the default dev env
- local Convex mode is optional and only for specific cases

### Coder agent

Update guidance to include:

- new deployment-aware workflow
- use `insights` for operational diagnosis
- mention pagination read caps
- clarify when to use query vs mutation vs action vs internal functions

### Convex skill

Update it to:

- include new deployment tools
- include env promotion/cloning patterns
- mention `.convex` local mode as optional
- include `maximumRowsRead` / `maximumBytesRead`

### Convex auth skill

Keep mostly as-is, but:

- mention local anonymous mode compatibility if adopted
- keep cloud-first auth flow as default

### dev-run tool

Keep the current role, but improve it:

- still handle sync + build + PM2 restart
- optionally understand deployment context
- optionally surface Convex sync failures in a more structured way

### write-client-env tool

Decide explicitly whether Surgent wants:

- `.env` as the canonical sandbox client env file
- `.env.local` as the canonical sandbox client env file

Current harness uses `.env`.
Convex CLI deployment selection updates `.env.local`.

This mismatch should be resolved intentionally.

## Important caveats

### Do not blindly adopt Convex AI files

Reason:

- Surgent already has a custom orchestrator pattern
- Surgent already has custom agent roles
- Surgent already has custom skills and tool conventions

Recommendation:

- test Convex AI files in generated app repos only
- do not let them become the top-level opencode instruction source
- if used, prefer opt-in and collision-aware installation

### The current config repo has stale documentation

Observed issues:

- README references `install.sh`, but repo ships `setup-opencode-config.sh`
- README's agent/model table is stale relative to actual files
- setup script references a `packages/opencode/.opencode` local override path that does not exist in the Surgent repo

This should be cleaned up so future agent behavior is easier to reason about.

## My recommendation

### Best immediate path

Keep Surgent cloud-Convex-first, but make it operationally smarter.

That means:

- keep `convex_create_project`
- keep embedded dashboard UX
- keep UI-first orchestration
- add health/deployment/env promotion MCP capabilities
- refresh opencode skills so agents actually know those capabilities exist

### What I would build first

If I were implementing this now, I would do these in order:

1. `insights` MCP tool
2. `call_action` MCP tool
3. deployment list/create MCP tools
4. opencode config repo instruction refresh
5. region-aware deployment creation

### What I would postpone

- making local anonymous Convex the default
- automatic Convex AI file installation

Those are promising, but they are second-order wins compared with the missing operational MCP surface.
